## 摘要
检索增强生成（RAG）通过将生成过程基于外部证据来提升大语言模型（LLM）的答案质量，但处理检索到的上下文使得预填充阶段成为主要的服务成本。RAG缓存融合通过重用检索到的块的预计算键值（KV）缓存，并在当前提示下选择性重新计算令牌来降低成本。然而，现有的选择器在质量和效率之间存在两难：快速的查询无关或最终层查询到上下文选择器可能遗漏与请求相关的证据，而全视图查询感知选择器在重新计算之前需要广泛的上下文和层可见性，因此会阻塞逐层的缓存融合流水线。我们提出QCFuse，一种用于RAG缓存融合的压缩视图查询感知选择器。QCFuse使用块锚点查询探测来将用户查询状态条件化到紧凑的每块锚点上，并使用关键层分析来识别重新计算令牌而无需检查所有层。我们在SGLang中实现QCFuse，并在六个数据集上的四个开放权重LLM上进行评估。QCFuse达到了完全预填充级别的质量。在相同质量下，QCFuse相比完全预填充平均预填充时间加速1.7倍，相比最强质量保持基线ProphetKV加速1.5倍。

## Abstract
Retrieval-augmented generation (RAG) improves large language model (LLM) answer quality by grounding generation in external evidence, but processing retrieved contexts makes the prefill stage a dominant serving cost. RAG cache fusion reduces this cost by reusing precomputed key-value (KV) caches for retrieved chunks and selectively recomputing tokens under the current prompt. Existing selectors, however, face a dilemma between quality and efficiency: fast query-agnostic or final-layer query-to-context selectors can miss request-relevant evidence, whereas full-view query-aware selectors require broad context and layer visibility before recomputation and therefore stall the layer-wise cache-fusion pipeline. We present QCFuse, a compressed-view query-aware selector for RAG cache fusion. QCFuse uses chunk-anchor query probing to condition user-query states on compact per-chunk anchors and critical-layer profiling to identify recomputation tokens without all-layer inspection. We implement QCFuse in SGLang and evaluate it on four open-weight LLMs across six datasets. QCFuse reaches full-prefill-level quality. At matched quality, QCFuse achieves an average prefill-time speedup of 1.7x over full prefill and 1.5x over ProphetKV, the strongest quality-preserving baseline.

- **核心问题**：为缓解预填充开销，业界提出了**KV缓存融合（KV Cache Fusion）** 技术，即重用已计算好的文本块KV缓存，并选择性地重新计算部分令牌以恢复上下文依赖性。然而，现有的令牌选择器面临**质量与效率的权衡**：
    - **查询无关或仅依赖最终层**的选择器（如CacheBlend, EPIC, FusionRAG）速度快，但会错过与用户查询相关的关键证据，导致生成质量下降。
    - **全视图查询感知**的选择器（如ProphetKV）通过加载全部上下文和所有层信号来精确选择令牌，生成质量高，但查询过程需要加载大量KV缓存，**阻塞**了层间缓存融合流水线，导致延迟增加。
- **本文目标**：旨在设计一种新的查询感知选择器，既能像全视图方法一样保证生成质量，又能保持类似快速方法的流水线效率，从而突破现有方法的权衡困境。

#### 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想：压缩视图（Compressed View）**：QCFuse的核心是**在保持查询感知能力的同时，大幅压缩选择器需要观测的上下文（Token View）和层（Layer View）信息量**，从而在不阻塞流水线的前提下获得高质量的选择结果。
- **关键技术细节**：
    - **挑战1: Token视图条件化瓶颈 (Challenge 1)**：全上下文探测需要加载大量KV缓存。
        - **解决方案：块锚点查询探测 (Chunk-Anchor Query Probing)**
            1.  **离线锚点构建**：在离线阶段，对每个可重用的文本块（Chunk），使用**KVzip**（一种基于自监督重建的排序规则）选出最具有代表性的部分令牌作为该块的“锚点”。默认保留10%的令牌作为锚点。
            2.  **在线锚点探测**：在线阶段，QCFuse不加载全部上下文，而是只加载这些已选定的、紧凑的锚点KV缓存（Anchor KV Cache）。用户查询只与这些锚点进行注意力计算，从而高效地获得条件化于检索证据的查询状态。
    - **挑战2: 层视图定位瓶颈 (Challenge 2)**：全层分析需要加载所有层的KV缓存。
        - **解决方案：关键层分析 (Critical-Layer Profiling)**
            1.  **离线关键层分析**：在离线阶段，使用校准数据集（如SQuAD, NewsQA），测试每一层单独进行令牌定位的准确率（Recall@10%），找出准确率最高的前3层作为“关键层”。
            2.  **在线仅用关键层**：在线阶段，选择器仅加载并利用这少数几个关键层的KV缓存来对原始上下文令牌进行重要性评分和选择，避免了全层扫描。
    - **算法流程**：
        1.  **离线准备**：预计算并存储每个文本块的PIC（位置无关缓存）KV缓存、KVzip选择的10%锚点KV缓存，以及通过分析确定的Top-3关键层集合。
        2.  **在线服务**：接收到用户请求后，首先加载锚点KV缓存和关键层KV缓存。然后，进行“块锚点查询探测”和“关键层令牌定位”，快速计算出需要重新计算的令牌集合P。最后，将P送入**流水线缓存融合**引擎，进行稀疏重计算并生成第一个令牌。
